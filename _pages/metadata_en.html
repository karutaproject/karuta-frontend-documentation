The metadata of an element allow the designer to give it a unique identifier and to define it's behavior.

The unique identifier is called the Semantic Tag and is used to reference an element from another element. The designer should attribute a unique identifier to every element added to a portfolio.

The next step is to specify the behavior of the element. For example, in the case of structural elements (Section, Page, Subsection), the designer may specify menus that allow the user to add elements to his or her portfolio. In the case of resources (the elements that contain the data), the designer can associate roles to each right, hence specifying who can do what on each element.

It is also in the metadata that the designer specifies which elements will populate a choice box, which users will be able to share their portfolio with another person (and under which security restrictions), create help bubbles, etc.

The metadata of an element is accessed through it's editing dialog box (pencil button).

Semantic Tag

An element's Semantic Tag is it's identifier and should express it's semantics (example: student-profile-page, student-name, etc). For each element the designer adds to a portfolio, he or she should replace the default semantic tag by one that expresses what the element's purpose is and what it will contain.

The semantic tag is used to search for an element in a portfolio. It is used in the Search metadata (see below) as well as in the report and batch scripts.

IMPORTANT: Search of a semantic tag is inclusive — for example, the search for all elements with semantic tag competency will also yield all elements with the semantic tag sub-competency, as the chain of characters of the first is included in the second.

Code

Most often attributed to Item and Subsection elements that will appear in single choice boxes (GetResource and GetGetResource elements) and multiple choice boxes (Get_Multiple and Import_Multiple functions). It is by the code that the user's selection is identified.

IMPORTANT: The code of a portfolio element must be unique within this portfolio.

By default, both the element's code and label are displayed in the choice box, and then only the label of the user's selection is displayed in the GetResource (or GetGetResource) element. This behavior may be modified by adding one or more of the following symbols to the element's code:

  • @ the code will not be displayed in the choice box
  • # the code will be displayed in the GetResource element
  • $ the label will not be displayed in the choice box
  • % the label will not be displayed in the GetResource element
  • ? the value will be displayed in the choice box (only Item elements have a field to indicate a value)
  • & the value will be displayed in the GetResource element (only Item elements have a field to indicate a value)
  • ! the element will not be selectable in the choice box (useful to create section titles in the choice box)

Multilingual Node

Specifies whether an element's label has a different value in each language (if the portfolio is multilingual). If Multilingual Node is not checked, there will only be one value for all languages, and that value will be stored in the default language (the default language is defined in the _init.js configuration file).

IMPORTANT: The code is never multilingual. It has the same value independently of the selected working language.

Multilingual Resource

Resource elements only. Specifies whether the data in the resource element has a different value in each language (if the portfolio is multilingual). If Multilingual Resource is not checked, there will only be one value for all languages, and that value will be stored in the default language (the default language is defined in the _init.js configuration file).

For example, a name or a birth date are usually not language dependant, whereas a text that gives instructions is definitely language specific.

Encrypt Resource

Resource elements only. Specifies whether the data in the resource element must be encrypted. If so, a key phrase is requested from the user when they open the portfolio. This phrase is used to encrypt/decrypt the data. It is not stored in the portfolio and thus must be memorized by the user, as he or she will be asked to provide it every time they open the portfolio.

See

List of roles, separated by a space character, that have the right to see the element. For example: student teacher.

By default, the right to see is attributed to the role all, which includes all roles defined on the portfolio. This is the only right to have a default value.

Delete

List of roles, separated by a space character, that have the right to delete the element. Displays the X button.

Edit Resource

Resource elements only. List of roles, separated by a space character, that have the right to edit the element (store and modify data in the element). Displays the pencil button.

Resource Without Pencil

Resource elements only. Does not display the pencil button to the roles that have the Edit Resource right. Used in combination with the Import_Multiple and Get_Multiple functions (see the MENU FONCTIONS page for details).

Comment

List of roles, separated by a space character, that have the right to comment the element. Adds a text editor at the bottom of the element's editing box. In the case of resource elements, the comment is timestamped and signed with the user's name. The user must also have the Edit Resource right on the element.

TIP: Designers can give themselves the Comment right on structural elements — this way they can add the equivalent of a subtitle to the element's label.

Submit

List of roles, separated by a space character, that have the right to submit the element. Upon submission the Edit Resource and Delete rights are removed from the element and all it's children. Displays the Submit button.

Edit Code

List of roles, separated by a space character, that have the right to modify the code of the element.

Edit Label

List of roles, separated by a space character, that have the right to modify the label of the element.

Node Without Pencil

Does not display the pencil button to the roles that have the Edit Label right. Used in combination with the Import_Multiple and Get_Multiple functions (see the MENU FONCTIONS page for details).

Duplicate

List of roles, separated by a space character, that have the right to duplicate the element. The new element will be added at the end of the list of the original element's siblings (the elements of the same hierarchical level in the tree). Displays the copy button.

Resize Image

Image elements only. List of roles, separated by a space character, that have the right to modify the size in which the image is displayed. Displays the pencil button.

Edit Proxy Target

Proxy elements only. List of roles, separated by a space character, that have the right to modify the element displayed by Proxy. The value of the original element is automatically updated.

Ordering

List of roles, separated by a space character, that have the right to modify the position of the element amongst it's siblings (the elements of the same hierarchical level in the tree). Displays the upward arrow button.

Move

List of roles, separated by a space character, that have the right to move the element to another branch of the portfolio tree. Displays the intersecting arrows button.

Move In

Semantic Tag that restricts the list of elements to which the element can be moved. Used in combination with the Move right. User can only move the element to an element that has the specified semantic tag (example: competency-page).

Show/Hide

List of roles, separated by a space character, that have the right to show/hide the element and all it's children to other users. Displays the eye button. The eye button is a toggle button and indicates the state, not the action: when the eye is not slashed, the element is visible to the other users.

Show To

List of roles, separated by a space character, to whom the element is shown or hidden by the users that have the Show/Hide right.

Print

List of roles, separated by a space character, that have the right to print the element. Displays the printer button.

Editing Box Title

Specifies an alternate title for the element's editing box (by default the title is the element's label). Useful for resource elements that don't have a label or to give instructions to users with the Edit Resource right.

Max Words

Long Text elements only. Specifies the maximum number of words that the user with the Edit Resource right can enter into the element.

Visible

When not selected, the element is not displayed (except to the designer). Useful to hide to users content that is needed for the execution of a report script. All elements are visible by default.

Inline Editing

Resource elements only. When selected, instead of displaying the pencil button to the user with the Edit Resource right, the resource's editor is displayed directly on the portfolio page.

Collapsible

Subsection elements only. When not selected, the +/- toggle button to the left, through which the subsection may be extended or collapsed, is not displayed. Subsections are collapsible by default.

Query

The Query metadata is used to specify which elements will be displayed in a choice box (GetResource and GetGetResource elements), as well as to specify the target of a Proxy element (see RESOURCES for details on GetResource, GetGetResource and Proxy elements).

Query Format for GetResource elements

portfolio code.semantic tag.label

  • portfolio code Code of the rubrics portfolio in which to search for the elements to display in the choice box. The format is project code.rubrics portfolio code. If project code is not present, the search for the rubrics portfolio will take place in the current project (project containing the GetResource element).
  • semantic tag Semantic tag of the elements to display in the choice box.
  • label Reserved word. When searching for Item or Subsection elements (see other options below).

Option self
Can replace portfolio code. In that case, the search will take place within the current portfolio (portfolio containing the GetResource element).

Option text
Can replace label if the elements searched for are Long Text or Short Text elements. In that case, it is the text in the resource that will be displayed in the choice box and then in the GetResource element.

Option fileid
Can replace label if the elements searched for are Image elements. In that case, it is the image in the resource that will be displayed in the choice box and then in the GetResource element. The Image elements must be created inside the same portfolio as the GetResource element; hence to be used in combination with the self option.

Option resource
Can replace label if the elements searched for are Item elements. In that case, the GetResource element will be updated if the selected Item element is modified. The Item elements must be created inside the same portfolio as the GetResource element; hence to be used in combination with the self option.

Query Format for GetGetResource elements

portfolio code.position.semantic tag1.semantic tag2.label

  • portfolio code Code of the rubrics portfolio in which to search for the elements specified by semantic tag2. The format is project code.rubrics portfolio code. If project code is not present, the search for the rubrics portfolio will take place in the current project (project containing the GetGetResource element).
  • position Indicates where in the current portfolio to search for the code of the element that will restrict the search in the Rubrics portfolio. Takes one of the following values child, sibling, parent, parent.parent, parent.parent.parent.
  • semantic tag1 Semantic tag of the parent element (restricts the search to the children of this parent).
  • semantic tag2 Semantic tag of the elements to display in the choice box.
  • label Reserved word.

Query Format for Proxy elements

portfolio code.semantic tag.label

  • portfolio code Code of the portfolio in which to search for the target to display.
  • semantic tag Semantic tag of the elements amongst which to choose the target to display.
  • label Reserved word.

Share

The Share metadata is used to allow users to send a third party an email containing a hyperlink that shares part or all of their portfolio with this person.

Share Format

role that shares,visitor role,role(s) and/or email(s) of recipient(s),security level,link lifespan,button label@en[,condition]

  • role that shares Role of user that can share (will have the sharing button).
  • visitor role Role the recipient will have in the portfolio.
  • role(s) and/or email(s) of recipient(s) Role(s) and/or email(s) of recipient(s). If it is a role, all users who have been shared with the portfolio in this role will receive an email.
  • security level 0 to 4 (see security level descriptions below).
  • link lifespan In hours. After this time, the link in the email will be deactivated.
  • button label Label that will appear on the sharing button. Append with @en for English.
  • condition Optionnal. Condition for the sharing button to be available to the user. Example: submitted=='Y'.

Option ?
Can replace role(s) and/or email(s) of recipient(s). In this case the email's recipient is not predetermined by the designer; the user will specify an email address in a pop-up window.

Security Levels

Level 0 – Karuta user
The email's recipient has a Karuta account and the portfolio is already shared with him or her in the appropriate role. Karuta will require their password if they are not already logged in.

Level 1
The email's recipient has a Karuta account and the portfolio is already shared with him or her in the appropriate role. Karuta will not require their password (if they are not already logged in, a temporary password will be created). For security purposes, there should be a Submit button on the elements the visitor can edit.

Level 2
The email's recipient has a Karuta account but the portfolio is not shared with them. Karuta will share the portfolio with the user in the appropriate role. If they are not already logged in, a temporary password will be created. For security purposes, there should be a Submit button on the elements the visitor can edit.

Level 3
If the email's recipient doesn't have a Karuta account, a temporary user ID will be created. Karuta will share the portfolio with the user in the appropriate role. For security purposes, there should be a Submit button on the elements the visitor can edit.

Level 4 – Public link
If the email's recipient doesn't have a Karuta account, a temporary user ID will be created. The user's role is all: can see but cannot edit. The portfolio instance must be shared with the user public in the role all.

Use Case Examples

The student can show his or her portfolio to anyone (Karuta user or not) in read-only mode (no edit rights):
student,all,?,4,unlimited,Share@en

The student can show his portfolio to anyone (Karuta user or not) in read-only mode (no edit rights), but the sharing has a limited duration of 24 hours:
student,all,?,4,24,Share@en

The student requests for his or her teacher to proceed with their evaluation; the teacher has a Karuta account and the student's portfolio is shared with said teacher in the teacher role:
student,teacher,teacher,1,12,Request an evaluation@en

The student requests for his or her internship supervisor to proceed with their evaluation; the internship supervisor does not have a Karuta account:
student,evaluator,yves.lacoste@airfrance.com,3,12,Request an evaluation@en

The student asks a peer to add a review in his or her portfolio; the colleague has a Karuta account:
student,peer,?,2,12,Request a comment@en

Menus

The Menu metadata is used to compose menus to import components into a portfolio. Usually the components are pre-built by the designer in a Reusable Components portfolio and the menu allows users to add these parts to their portoflio. In this case the imported component is copied and then pasted into the portfolio.

TIP: Designers can create menus for themselves to copy portfolios parts from one portfolio model to another.

Karuta also has various pre-programmed functions that can be called through the Menu metadata. See the MENU FONCTIONS page for complete specification.

Menu Format

portfolio code,semantic tag,menu label@en,role(s)[;other menus]

  • portfolio code Code of the portfolio from which to import the component
  • semantic tag Semantic tag of the component to import
  • menu label Label that will appear on the menu button. Append with @en for English.
  • role(s) List of roles, separated by a space character, that will have the menu element.

NOTE : There is no final semicolon.

Menu Button Text

Alternate text for the Menu button when more than one option is offered to a role (by default Menu). Append with @en for English. Example: Add a diploma@en if there are multiple types of diploma components.

Help Bubble

Displays additional information in a pop-up bubble next to the element's label. Append with @en for English.
