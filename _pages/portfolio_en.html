In Karuta portfolios are stored as structured trees. All portfolios are built following an XML tree structure (see the STRUCTURES page for more detail).

There are multiple types of Karuta trees:

  • Portfolio Model
  • Portfolio Instance
  • Rubrics Portfolio
  • Reusable Components Portfolio
  • Report Script
  • Batch Script
  • Batch Form

Portfolio Model

A portfolio model is a tree structure that organizes resources. Resources are the elements that will contain data: data supplied to the users (like instructions or files), and data supplied by the users to populate their portfolios. Karuta offers resources of many types: text field, text editor, file, image, audio and video player, calendar, etc (see the RESOURCES page for more detail).

Once built by the designer, the portfolio model is instantiated for each user, which means that a copy of the model is created and the new tree is the portfolio for that specific user.

Portfolio Instance

A portfolio instance is a copy of the model for a specific user. At the moment of instantiation (copying) the roles and rights are created for each element of the portfolio, according to what the designer has indicated in the metadata (see the METADATA page for more detail).

Rubrics Portfolio

A rubrics portfolio contains lists of elements that will be referenced in other portfolios, for example a competency framework or the choices of a dropdown list.

Reusable Components Portfolio

A reusable components portfolio is used to store portfolio parts that will then be made available to users through a menu. This allows users to add parts to their portfolio as needed.

Report Script

A report is a script that contains specific commands to select elements from one or multiple portfolios and display them in a different layout. For exemple to build dashboards, resumes or cohort aggregate reports. The report script commands are used to navigate through portfolios to select the elements to be displayed. The result is then displayed in an HTML table, the layout of which is customizable by the designer (see the Report page for more detail).

Batch Script

A batch is a script that contains specific commands to automate the process of creating user accounts, creating portfolio instances, updating information in these portfolios, etc. A batch script is usually associated to a CSV file containing, for example, the list of users. Executing the batch script with this file will then automatically create the user accounts and the portfolios (see the Batch page for more detail). A batch script may also be associated with a batch form.

Batch Form

A batch form executes a batch script for one single user, without the need for a CSV file.

How It Works

A Karuta tree or portfolio is built by adding branches to the root element, much like the assembly of Lego blocks. Certain branches are predefined: the basic Karuta elements. Other branches are built by the designer by combining these basic elements. In this manner complex portfolios are built.

User menus may also be defined, allowing the end user to add pre-built branches to their portfolio as need be.

Model and Instance

There are two types of Karuta portfolios:

  • Portfolio model – Built by the designer. The associated roles are designer and all.
  • Portfolio instance – Instantiated from a portfolio model for each individual user. The associated roles are those defined by the designer in the model, for example student and teacher, plus designer and all.

The Security Model

  • A portfolio has associated roles (example student, teacher, etc).
  • A role has associated rights (read, edit, submit, etc) for each of the portfolio elements.
  • Sharing a portfolio is done by indicating a user and a role.

Simulation of the Roles by the Designer

In the portfolio model, the designer may (and should!) simulate the various roles he or she has created to verify the rights attributed on each element.
